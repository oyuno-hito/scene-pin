# パーソナル動画ブックマーク＆プレイヤー 要求仕様書（サーバー構成版）

## 1. 概要

本バージョンでは、v1.0.0のフロントエンド完結型アーキテクチャから、ローカルサーバーを利用したクライアント・サーバー構成へ移行する。これにより、デバイスのストレージ制約から解放され、外付けHDDやNAS等の大容量ストレージを活用した動画管理が可能となる。

### 1.1 v1.0.0からの主な変更点

- 動画ファイルの保存先をサーバー側ストレージへ移行
- メタデータ管理をPostgreSQLへ移行
- ストレージ抽象化によるバックエンド柔軟性の確保（将来的なNAS/S3/Google Drive対応を見据える）

## 2. 機能要件

### 2.1 動画再生機能（既存機能の継続）

* **動画取得:** サーバーAPIから動画をストリーミング再生する。
* **再生制御:** 再生/一時停止、シーク、ボリューム調整。
* **再生速度調整:** 0.5x 〜 2.0x の範囲で動的に変更可能。
* **視聴位置の自動保存:** 一時停止・画面離脱時に視聴位置をクライアント側（IndexedDB）に保存し、再開時に復元する。

### 2.2 ブックマーク（ピン）機能（既存機能の継続）

* **タイムスタンプ保存:** 再生中の任意位置にマーカー（ピン）を打つ。
* **編集機能:** 保存したタイムスタンプに対するメモの編集・削除。
* **ジャンプ機能:** リスト化されたタイムスタンプをタップし、該当箇所へ瞬時にシーク。

### 2.3 ループ再生機能（既存機能の継続）

* **A-B間ループ:** 指定した開始点と終了点の間をリピート再生する。

### 2.4 動画管理機能（新規）

* **動画アップロード:** クライアントから動画ファイルをサーバーにアップロードする。
* **サムネイル自動生成:** アップロード時にサーバー側でサムネイルを自動生成・保存する。
* **動画削除:** 動画ファイルと関連メタデータ（ブックマーク含む）をCASCADEで削除する。
* **タグ管理:** 動画に対してタグを付与し、分類・検索を容易にする。

### 2.5 検索・一覧機能（新規）

* **動画一覧取得:** サムネイル、動画名、タグを含む一覧を取得する。
* **タグフィルター:** 指定したタグで動画を絞り込む。
* **ソート:** 追加日時、更新日時、動画名でソート可能。

## 3. API仕様

### 3.1 動画本体管理API

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| POST | `/api/videos/upload` | 動画ファイルのアップロード（サムネイル自動生成） |
| GET | `/api/videos/{id}/stream` | 動画ストリーミング（Range対応） |
| DELETE | `/api/videos/{id}` | 動画削除（メタデータもCASCADE削除） |

### 3.2 動画メタデータ管理API

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | `/api/videos/{id}` | 動画メタデータ取得 |
| PATCH | `/api/videos/{id}` | 動画メタデータ更新（名前、タグ等） |
| GET | `/api/videos/{id}/thumbnail` | サムネイル画像取得 |

> **Note:** 視聴位置はクライアント側（IndexedDB等）で管理するため、サーバーAPIは不要。

### 3.3 ブックマーク管理API

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | `/api/videos/{id}/bookmarks` | 動画のブックマーク一覧取得 |
| POST | `/api/videos/{id}/bookmarks` | ブックマーク追加 |
| PATCH | `/api/bookmarks/{id}` | ブックマーク更新（メモ編集） |
| DELETE | `/api/bookmarks/{id}` | ブックマーク削除 |

### 3.4 検索・一覧API

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | `/api/videos` | 動画一覧取得（クエリパラメータでフィルター/ソート） |
| GET | `/api/tags` | タグ一覧取得（タグ候補表示用） |

#### クエリパラメータ例

```
GET /api/videos?tags=dance,practice&sort=updatedAt&order=desc&page=1&limit=20
```

### 3.5 追加提案API

| メソッド | エンドポイント | 説明 |
|---------|---------------|------|
| GET | `/api/health` | ヘルスチェック（サーバー死活監視用） |
| GET | `/api/storage/status` | ストレージ使用状況（残容量確認用） |

## 4. データモデル

### 4.3 tags テーブル

| カラム | 型 | 説明 |
|--------|-----|------|
| id | BIGSERIAL | 主キー |

### 4.4 video_tags テーブル（中間テーブル）

| カラム | 型 | 説明 |

## 5. 非機能要件

* **プラットフォーム:** iOS Safari（Capacitorネイティブアプリも継続サポート）
* **データ永続化:** PostgreSQLでメタデータ管理、ファイルストレージで動画保存
* **ストリーミング:** HTTP Range Requestsに対応し、シーク時の部分取得を可能にする
* **認証認可:** ローカルネットワーク内での利用を前提とし、当面は実装しない
* **ストレージ抽象化:** 将来的なNAS/S3/Google Drive対応を見据え、StorageAdapterパターンを採用

## 8. デプロイ構成

### 8.1 概要

ローカルサーバー（PC + 外付けHDD）にDocker Composeでアプリケーションを構築し、GitHub Container Registry (ghcr.io) へのイメージpushをトリガーにWatchtowerで自動更新する。

### 8.2 構成図

```
┌─────────────────────────────────────────────────────┐
│ GitHub                                              │
│  ┌─────────────┐      ┌──────────────────────┐     │
│  │ Repository  │──────│ GitHub Actions       │     │
│  └─────────────┘      │  - Build jar         │     │
│                       │  - Build Docker image│     │
│                       │  - Push to ghcr.io   │     │
│                       └──────────┬───────────┘     │
└──────────────────────────────────┼─────────────────┘
                                   │
                    ┌──────────────▼───────────────┐
                    │ ghcr.io/you/scene-pin-server │
                    └──────────────┬───────────────┘

┌──────────────────────────────────┼─────────────────┐
│ ローカルサーバー                  │                 │
│  ┌────────────┐    ┌─────────────▼─────────────┐  │

│                    └───────────────────────────┘  │
└────────────────────────────────────────────────────┘
```

### 8.3 GitHub Actions ワークフロー

```yaml
name: Build & Push Docker Image

on:
  push:
    branches: [main]
    paths:
      - "server/**"

jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v4
      
      - uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'
      
      - name: Build with Gradle
        working-directory: server
        run: ./gradlew bootJar
      
      - name: Login to GitHub Container Registry
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: server
          push: true
          tags: ghcr.io/${{ github.repository_owner }}/scene-pin-server:latest
```

### 8.4 Dockerfile（server/Dockerfile）

```dockerfile
FROM eclipse-temurin:21-jre-alpine

WORKDIR /app

COPY build/libs/*.jar app.jar

EXPOSE 8080

ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 8.5 Docker Compose（ローカルサーバー）

```yaml
version: '3.8'

services:
  app:
    image: ghcr.io/your-username/scene-pin-server:latest
    ports:
      - "8080:8080"
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://postgres:5432/scenepin
      - SPRING_DATASOURCE_USERNAME=scenepin
      - SPRING_DATASOURCE_PASSWORD=scenepin
      - STORAGE_PATH=/data/videos
      - THUMBNAIL_PATH=/data/thumbnails
    volumes:
      - /mnt/external-hdd/scene-pin/videos:/data/videos
      - /mnt/external-hdd/scene-pin/thumbnails:/data/thumbnails
    depends_on:
      - postgres
    restart: unless-stopped

  postgres:
    image: postgres:16-alpine
    environment:
      - POSTGRES_DB=scenepin
      - POSTGRES_USER=scenepin
      - POSTGRES_PASSWORD=scenepin
    volumes:
      - /mnt/external-hdd/scene-pin/postgres-data:/var/lib/postgresql/data
    restart: unless-stopped

  watchtower:
    image: containrrr/watchtower
    volumes:
      - /var/run/docker.sock:/var/run/docker.sock
      - ~/.docker/config.json:/config.json:ro
    environment:
      - WATCHTOWER_CLEANUP=true
      - WATCHTOWER_POLL_INTERVAL=300
      - WATCHTOWER_INCLUDE_RESTARTING=true
    restart: unless-stopped
```

### 8.6 マイグレーション

Spring Boot + Flywayを使用し、アプリケーション起動時に自動でマイグレーションを実行する。

```
server/src/main/resources/db/migration/
  ├── V1__create_videos_table.sql
  ├── V2__create_bookmarks_table.sql
  └── V3__create_tags_tables.sql
```

### 8.7 初回セットアップ手順

```bash
# 1. 外付けHDDのマウント確認
mount | grep external-hdd

# 2. ディレクトリ作成
sudo mkdir -p /mnt/external-hdd/scene-pin/{videos,thumbnails,postgres-data}

# 3. ghcr.io へのログイン（Personal Access Token必要）
echo $GITHUB_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# 4. Docker Compose起動
docker compose up -d

# 5. ログ確認
docker compose logs -f app
```

### 8.8 更新フロー

1. `main`ブランチにpush
2. GitHub Actionsがjarビルド → Dockerイメージビルド → ghcr.ioにpush
3. Watchtowerが5分ごとにイメージ更新をチェック
4. 新しいイメージがあれば自動でpull → コンテナ再起動
5. Flywayが起動時にマイグレーションを実行

### 8.9 ストレージ使用量の目安

| 項目 | サイズ |
|------|--------|
| app イメージ（JRE Alpine + jar） | 約150-200MB |
| postgres イメージ | 約80MB |
| watchtower イメージ | 約10MB |
| **合計（初回pull）** | **約250-300MB** |
| 更新時（レイヤーキャッシュ有効） | 数MB〜数十MB |

## 9. 開発ロードマップ

| フェーズ | 内容 |
|---------|------|
| **Phase 1: サーバー基盤構築** | Spring Boot プロジェクト作成、PostgreSQL接続、基本API実装 |
| **Phase 2: 動画管理API** | アップロード、ストリーミング、サムネイル生成 |
| **Phase 3: メタデータ・ブックマークAPI** | CRUD API、タグ機能 |
| **Phase 4: フロントエンド移行** | API連携、IndexedDB依存の除去 |
| **Phase 5: ストレージ抽象化** | StorageAdapterパターン導入、将来の拡張準備 |

## 9. 運用・拡張方針

* **ローカルサーバー運用:** Tailscale経由で宅外からもセキュアにアクセス可能
* **ストレージ拡張:** StorageAdapterの差し替えにより、NAS/S3/Google Driveへの移行が容易
* **認証追加:** 将来的に外部公開する場合は、Spring Securityによる認証を追加可能

### 補足：v1.0.0からの移行パス

既存のCapacitorアプリ（v1.0.0）からの移行は以下の手順で行う：

1. サーバー環境の構築・起動
2. フロントエンドの環境変数に `VITE_API_BASE_URL` を設定
3. 既存のIndexedDBデータは手動またはマイグレーションスクリプトで移行（必要に応じて）
4. Capacitorアプリの再ビルド・再インストール
