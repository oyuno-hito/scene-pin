# 開発ワークフロー

## 要求仕様書の判定

各仕様書のフロントマター `status` で状態を判定する。

### ステータス定義

| status | 意味 | 同時に存在できる数 |
|--------|------|------------------|
| `released` | 実装済み | 複数可 |
| `in-progress` | 現在実装中 | **1つのみ** |
| `planned` | 計画段階 | 複数可 |

### 新しいチャットでの判定フロー

1. `docs/要求仕様書/` 配下の全ファイルを確認
2. `status: in-progress` があれば → その仕様書に従って継続
3. なければ → ユーザーに「どの仕様書を実装しますか？」と確認
4. すべて `released` → 新機能なら新規仕様書の作成を提案

### 要求仕様書がない場合

- 新機能や大きな変更の場合、先に仕様書を作成
- ユーザーと内容を確認してから実装に着手

## TODOの消化フロー

実装時は以下のフローで進める:

1. **TODOリストを作成する**
   - 作業を適切な粒度に分割
   - 各TODOは独立してコミット可能な単位

2. **TODOを1つずつ消化する**
   - 1つのTODOを完了したら、次に進む前に**ユーザーの確認を待つ**
   - ユーザーがコミットまたは確認の返答をしてから次のTODOへ

3. **インタラクティブな修正を受け入れる**
   - ユーザーが手動で修正を行った場合、その方針を確認
   - 方針の変更があれば、以降のTODOに反映

## 確認ポイント

各TODO完了時に以下を確認:
- ビルドが通ること
- ユーザーの意図と合っていること
- 方針変更があれば記録すること

## 要求仕様書の同期更新

**実装中に仕様が変更された場合、要求仕様書も即座に更新すること。**

### 更新が必要なケース

1. **データモデルの変更**
   - カラムの追加・削除・型変更
   - テーブル構造の変更

2. **技術スタックの変更**
   - ORM、フレームワーク、ライブラリの変更

3. **API仕様の変更**
   - エンドポイント、リクエスト/レスポンス形式の変更

4. **機能の追加・削除・変更**
   - ユーザーの指示で機能が変わった場合

### 更新の原則

- 要求仕様書は「現在の実装目標」を反映する
- 実装と仕様書の乖離を放置しない
- 変更履歴はgitで管理（仕様書内に変更履歴セクションは不要）
